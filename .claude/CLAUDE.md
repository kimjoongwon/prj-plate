# 프로젝트 개발 가이드

## 프론트엔드 개발 규칙

### 컴포넌트 작성
- shared-frontend 컴포넌트를 만들 때는 mobx를 사용합니다
- 이벤트 핸들러 함수명에는 `handle` 접두어를 붙입니다 (예: `handleClick`, `handleChange`)
- 컴포넌트 내에서 함수를 인라인으로 선언하지 않습니다

### 텍스트 처리
- 모든 컴포넌트의 텍스트는 components 폴더의 Text 컴포넌트로 감싸서 사용합니다

## 테스트 작성 규칙

- 테스트 코드의 설명(describe, it)은 한글로 작성합니다
- 테스트는 Given-When-Then 패턴을 따릅니다

## 커밋 메시지 규칙

```
<타입>(<범위>): <제목>

<본문>

<푸터>
```

### 타입
- feat: 새로운 기능
- fix: 버그 수정
- docs: 문서 수정
- style: 코드 포맷팅
- refactor: 코드 리팩토링
- test: 테스트 추가/수정
- chore: 빌드 작업, 패키지 매니저 설정 등

### 예시
```
feat(coin): 멀티시그 지갑 서비스 초기 구현

- 지갑 생성 API 추가
- 트랜잭션 승인 로직 구현
- 테스트 코드 작성
```

## 작업 위임 프로토콜

Task 도구나 subAgent를 사용하기 전에 **반드시** 다음 절차를 따릅니다:

### 1단계: 분석
사용자 요청을 분석하여 다음을 결정합니다:
- 필요한 subAgent 종류 (Explore, Plan, general-purpose 등)
- 작업 복잡도 (단순/중간/복잡)
- 분석 깊이 (기본/심층/최대)
- 예상 작업 범위 (파일 수, 영향 범위)

### 2단계: 제안
다음 형식으로 사용자에게 제안합니다:

📋 **작업 분석 완료**

**선택된 전략:**
- SubAgent: [Explore / Plan / general-purpose]
- 분석 깊이: [기본 / --think / --think-hard / --ultrathink]
- 작업 범위: [단일 파일 / 모듈 / 시스템 전체]

**선택 이유:**
[왜 이 전략을 선택했는지 간단히 설명]

**예상 작업:**
1. [구체적인 작업 1]
2. [구체적인 작업 2]
3. ...

이렇게 진행하시겠습니까?

### 3단계: 승인 대기
사용자의 명시적 승인을 받습니다:
- ✅ 승인: 제안대로 진행
- ✏️ 수정: 사용자가 전략/범위 조정
- ❌ 거부: 다른 접근 방법 모색

### 4단계: 실행
승인받은 전략대로 작업을 진행합니다.

### 예외 사항
다음의 경우 이 프로토콜을 생략할 수 있습니다:
- 사용자가 명시적으로 특정 subAgent 사용을 지시한 경우
- 이미 승인받은 작업의 연속 단계인 경우
- 단순 정보 조회나 파일 읽기만 필요한 경우
